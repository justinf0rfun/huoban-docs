# Checkpoint 对象

`Checkpoint` 是 Huoban 的校对点。它表示 runtime 不能或不应该继续自动执行，需要人、规则或 eval 给出明确结果。

规范页定义 Checkpoint 的字段形状和验证边界。为什么 checkpoint 是信任边界，请读 [Checkpoint 模型](/checkpoint-model)。

## 对象职责

Checkpoint 解决的问题是：高风险或高判断力节点不能只靠聊天里的“可以继续”。它必须成为对象，绑定 Run、stage、reason、允许动作和待审查产物。

Checkpoint 负责描述：

- 对应哪个 Run。
- 发生在哪个 stage。
- 为什么暂停。
- 允许哪些决策动作。
- 审查哪些 Artifact。
- 是否有过期时间。
- 决策状态。

Checkpoint 不负责：

- 声明所有 checkpoint 位置；这属于 `Flow.checkpoints`。
- 定义权限规则；这属于 `Policy`。
- 保存产物正文；这属于 `Artifact`。
- 替代人工判断；它只是把判断边界对象化。

## v1alpha1 字段

| 字段 | 要求 | 说明 |
| --- | --- | --- |
| `spec.runRef` | 必填 | 对应的 Run 引用。 |
| `spec.stage` | 可选 | 对应 stage id。 |
| `spec.reason` | 必填 | 触发原因。 |
| `spec.requiredActions` | 可选 | 允许动作：`approve`、`requestChanges`、`reject`。 |
| `spec.artifactRefs` | 可选 | 待审查 Artifact 引用。 |
| `spec.expiresAt` | 可选 | 过期时间字符串。 |
| `status.phase` | 可选 | `Pending`、`Approved`、`Rejected` 等实现侧状态。 |
| `status.conditions` | 可选 | 细粒度状态。 |

`requiredActions` 是允许动作列表，不是最终决策。最终决策应进入 `status.phase` 或 condition，并同步到相关 Run 状态。

## 最小合法示例

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Checkpoint
metadata:
  name: idea-to-spec-review-001-direction
spec:
  runRef:
    name: idea-to-spec-review-001
  stage: sharpen-idea
  reason: ConfirmDirectionBeforeSpec
  requiredActions:
    - approve
    - requestChanges
    - reject
  artifactRefs:
    - name: review-notes
```

参考 example：`examples/checkpoints/idea-to-spec-review-direction.yaml`。

## 验证规则

validator 至少应检查：

- `kind` 必须是 `Checkpoint`。
- `spec.runRef.name` 必须存在。
- `spec.reason` 必须存在。
- `requiredActions` 只能包含 `approve`、`requestChanges`、`reject`，且不能重复。
- `artifactRefs[]` 必须是 Artifact 引用形状。
- 如果 `stage` 存在，workspace validator 应确认它能解析到对应 Run 的 Flow stage。

## 与相邻对象的边界

| 容易混淆 | 边界 |
| --- | --- |
| `Checkpoint` / `Flow.checkpoints` | Flow 声明可能停在哪里；Checkpoint 对象记录一次具体停顿。 |
| `Checkpoint` / `Policy` | Policy 可以要求审批；Checkpoint 承载审批实例。 |
| `Checkpoint` / `Artifact.review` | Checkpoint 是决策边界；Artifact.review 是产物审查状态。 |
| `Checkpoint` / `Run.status` | Run 可以显示等待 checkpoint；Checkpoint 保存决策对象。 |

## 常见错误

- 只在聊天里同意继续，不创建 Checkpoint。这样 Run 无法审计。
- 在 Checkpoint 里写权限规则。权限应写在 Policy。
- 审查对象没有 Artifact 引用。高判断力产物应先成为 Artifact。
- 允许动作太宽泛。v1alpha1 只标准化三种动作。

## 活字印刷位置

Checkpoint 对应“校对”。印前、印中或付印前的关键判断必须显式留下校对记录；Checkpoint 让 Huoban 可控地试印和修版。
