# Policy 对象

`Policy` 是 Huoban 的印坊规矩：它决定什么允许、什么拒绝、什么必须经过审批。

规范页定义 Policy 的字段形状和验证边界。为什么 Policy 必须和 Profile 分离，请读 [Policy 模型](/policy-model)。

## 对象职责

Policy 解决的问题是：AI 工作的副作用不能只靠工具默认弹窗、口头约定或自然语言提醒。读文件、写文件、执行命令、访问网络、花钱、发布远端变更、访问密钥都必须有可审查规则。

Policy 负责描述：

- 默认如何处理未知副作用。
- 哪些 capability、side effect 或 trust level 匹配某条规则。
- 匹配后是 `allow`、`deny` 还是 `requireApproval`。

Policy 不负责：

- 描述项目偏好；这属于 `Profile`。
- 记录实际发生的副作用；这属于 `Run.status.observedSideEffects`。
- 执行审批动作；这属于 `Checkpoint`。
- 证明对象可信；这属于 `Trust`。

## v1alpha1 字段

| 字段 | 要求 | 说明 |
| --- | --- | --- |
| `spec.defaults` | 可选 | 默认策略。 |
| `spec.defaults.unknownSideEffect` | 可选 | 未声明副作用的处理：`allow`、`deny`、`requireApproval`。 |
| `spec.rules` | 必填，至少 1 条 | 策略规则数组。 |
| `spec.rules[].effect` | 必填 | `allow`、`deny` 或 `requireApproval`。 |
| `spec.rules[].when` | 必填，至少 1 个条件 | 匹配条件。 |
| `spec.rules[].when.capabilities` | 可选 | capability 数组。 |
| `spec.rules[].when.sideEffects` | 可选 | side effect 数组。 |
| `spec.rules[].when.trustLevels` | 可选 | trust level 字符串数组。 |
| `status` | 可选 | validation、derivation 或 review 状态。 |

v1alpha1 不支持命令级 DSL、路径匹配 DSL、正则匹配、模型匹配或 artifact type 匹配。这些可以在实现侧使用，但不能写成当前公共 schema 字段。

## 最小合法示例

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Policy
metadata:
  name: safe-defaults
spec:
  defaults:
    unknownSideEffect: requireApproval
  rules:
    - effect: requireApproval
      when:
        sideEffects:
          - publishRemote
          - spendMoney
          - accessSecret
    - effect: deny
      when:
        sideEffects:
          - modifyGitHistory
```

参考 example：`examples/policies/safe-defaults.yaml`。

## 验证规则

validator 至少应检查：

- `kind` 必须是 `Policy`。
- `spec.rules` 至少包含一条规则。
- 每条规则必须有 `effect` 和 `when`。
- `effect` 只能是 `allow`、`deny`、`requireApproval`。
- `when` 至少包含 `capabilities`、`sideEffects` 或 `trustLevels` 之一。
- `sideEffects` 只能使用标准 side effect 枚举。
- `defaults.unknownSideEffect` 如果存在，只能是 `allow`、`deny`、`requireApproval`。

## 规则解释建议

v1alpha1 不强制 rule ordering，但实现者应保持解释结果稳定。推荐：

- `deny` 优先于 `requireApproval`。
- `requireApproval` 优先于 `allow`。
- 未匹配任何规则时，使用 `defaults.unknownSideEffect`。
- 如果没有默认值，runtime 应保守处理，并写入 condition。

这些是解释建议，不是新字段。

## 与相邻对象的边界

| 容易混淆 | 边界 |
| --- | --- |
| `Policy` / `Profile` | Profile 是上下文；Policy 是权限和副作用边界。 |
| `Policy` / `Skill.sideEffects` | Skill 声明可能副作用；Policy 决定如何处理。 |
| `Policy` / `Checkpoint` | Policy 可以要求审批；Checkpoint 记录审批实例。 |
| `Policy` / `Trust` | Trust 描述来源信任；Policy 使用信任等级做规则判断。 |

## 常见错误

- 按具体命令写规则，例如 `git push`。v1alpha1 应按 side effect `publishRemote` 表达。
- 把 Policy 写成自然语言建议。规则必须可机器匹配。
- 以为 declared side effects 就是真实事实。实际发生的副作用要进入 `Run.status.observedSideEffects`。
- 在 Profile 里写权限约束但没有 Policy。这样无法稳定执行。

## 活字印刷位置

Policy 对应“印坊规矩”。它不决定印什么，也不决定如何排版；它决定哪些工序能自动做、哪些必须停下来校对、哪些直接禁止。
