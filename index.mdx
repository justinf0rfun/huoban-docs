Huoban 定义 AI 原生工作的**对象语言**，让 AI 工作 **可排版、可润墨、可印刷**。

Huoban 的最小不可替代价值是：把一次 AI 工作拆成**可验证、可组合、可审查**的标准对象，而不是绑定在某个 agent、聊天记录或私有 workflow 里。它用十个顶层对象描述能力、上下文、编排、执行、校对、治理、适配、产物、索引与信任，并用 Requirement、Preflight 与 Lifecycle Handler 补齐执行前依赖和生命周期观察，让不同 agent、tool、MCP server 和项目上下文共享同一套编排语义。

今天的 skill、MCP tool、AGENTS.md、CLAUDE.md、Cursor Rules、prompt、workflow 和执行记录分散在不同工具里。Huoban 把这些资产收敛为标准对象，让 AI 工作可以被描述、复用、迁移和治理。

如果用当前行业语言描述，Huoban 处理的是 **context engineering**、**skill reuse** 和 **agent loop standardization** 的交叉问题：上下文如何结构化，能力如何复用，循环如何在 `Checkpoint`、`Policy` 和 `Artifact` 中留下可审查记录。

Huoban 活板的设计哲学来自**活字印刷术**。它把 `Skill` 理解为活字，把 `Profile` 理解为墨方，把 `Flow` 理解为版式，把 `Run` 理解为锁版印刷，把 `Checkpoint` 理解为校对，把 `Policy` 理解为印坊规矩。

## 核心对象

| Object | 隐喻 | 标准职责 |
| --- | --- | --- |
| `Skill` | 活字 | 声明可复用能力、输入输出和副作用。 |
| `Profile` | 墨方 | 声明分层上下文、项目约束和团队规则。 |
| `Flow` | 版式 | 声明 capability-first 的编排、阶段、停顿和产物。 |
| `Run` | 锁版印刷 | 锁定一次执行的对象引用、mode、bindings，并可记录 `flowSpecHash`、`profileRefHash`、`policyRefHash` 与 `bindingHash`。 |
| `Checkpoint` | 校对 | 声明人类或系统必须判断的决策边界。 |
| `Policy` | 印坊规矩 | 声明权限、副作用、审批和危险动作限制。 |
| `Adapter` | 转接件 | 描述外部 skill、MCP tool、导入格式或外部能力如何进入 Huoban。 |
| `Artifact` | 印张 | 记录工作流产出的文档、报告、代码或决策。 |
| `Registry` | 字架 | 组织对象来源和索引范围。 |
| `Trust` | 来源信任记录 | 记录对象来源、审查、信任等级和 sandbox 要求。 |

## 共享执行契约

Requirement、Preflight Report 与 Lifecycle Handler 不是新的顶层对象。它们是嵌入十个对象体系的共享结构：

## 验证标准对象

Huoban 的入口不是运行某个特定 agent，而是验证标准对象、示例和解释器是否一致。

```bash
npm install
npm test
npm run validate
npm run explain
npm run print:dry-run
npm run preflight
```

如果你想看完整对象如何协作，从 [idea-to-spec-review 示例](/example-idea-to-spec-review) 开始，再运行 `npm run explain` 与 `npm run preflight`。
