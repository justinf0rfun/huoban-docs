`Run` 是 Huoban 的锁版印刷。`Flow` 描述可复用版式；`Run` 把一次工作实际采用的 Flow、动态输入、Profile、Policy、mode 与实现 bindings 固定下来，并为 Preflight、stage 状态、生命周期投递和 Artifact 提供统一索引。

Run 不是聊天记录，也不是完整执行日志。它保存足以解释“这一次准备如何工作、使用了什么、当前观察到什么”的声明和状态。

本页解释设计边界；字段级定义见 [Run 规范](/spec-run)。

## Run 的职责

Run 必须回答：

- 采用哪个 `Flow` generation？
- 本次使用哪些 `Profile` 与 `Policy` 引用？
- 当前是 explain、dryRun 还是 execute？
- Flow 的每个入口槽在这次调用中绑定了什么？
- 每个 stage 和 lifecycle role 绑定到哪个 `Skill` 或 `Adapter`？
- 这些声明对应哪些 snapshot hash？
- Run scope 的 Preflight 是否仍有效？
- 当前 stage、条件、副作用、产物与 lifecycle invocation 是什么？

Run 不负责定义可复用流程、权限规则、具体校对请求或产物正文。这些分别属于 `Flow`、`Policy`、`Checkpoint` 和 `Artifact`。

## 三种 mode

| mode | 目标 | 允许的结果 | 不应发生 |
| --- | --- | --- | --- |
| `explain` | 展开对象、引用和 binding 关系。 | 解释、诊断、`Unknown` readiness。 | 执行业务能力或产生副作用。 |
| `dryRun` | 生成可审查的计划态 Run 与 snapshot。 | 计划、bindings、Preflight 证据。 | 把计划误写成已完成执行。 |
| `execute` | 交给实现执行锁定的工作。 | stage 状态、Artifact、Checkpoint、lifecycle invocation。 | 在没有 Run scope `PreflightReady=True` 时启动。 |

schema 允许省略 `mode`，但 reference CLI 与 Preflight 按 `execute` 解释省略值。这样缺失字段不会形成 readiness 门禁旁路；需要解释或演练时必须显式写出 `explain` 或 `dryRun`。

dry-run 不等于 Preflight。dry-run 形成“准备执行什么”的 Run；Preflight 记录“当前环境的机械就绪检查结果”；Policy、Trust 与 Checkpoint 再回答“是否允许执行”。

## Input binding：从入口槽到本次调用

Flow 先声明入口契约：

```yaml
# Flow spec fragment
inputs:
  - name: document
    type: content
    required: true
```

Run 再锁定这一次的值：

```yaml
inputBindings:
  document:
    type: content
    contentRef:
      type: file
      path: requirements.md
    hash: sha256:...
```

支持的 binding 如下：

| 类型 | Run 保存内容 | 完整性语义 |
| --- | --- | --- |
| `content` | file、URL 或 Git blob `contentRef`，可选 `mediaType` | file 必须锁 raw-byte hash；远程载体在物化并验证前不能 ready。 |
| `text` | inline `value` | 完整 binding 由 `inputBindingHash` 锁定。 |
| `object` | inline object `value` | 使用 canonical JSON 进入 `inputBindingHash`。 |
| `objectRef` | typed object ref 与目标对象 `spec` hash | Flow 用 `objectKinds` 限定允许的 kind；Artifact 只是可能的目标之一。 |

这里刻意把**语义**与**载体**分开：Flow 接受 `content`，一次 Run 再选择 file、URL 或 Git blob。slash command、MCP、CLI 或其他入口都只是创建同一 binding 的实现方式，不进入公共协议。

协议不规定 `.huoban/runs/...` 之类的磁盘布局。实现可以把源文件复制到 Run 自己的内容寻址存储，但必须保存稳定内容引用与 hash。reference CLI 不获取 URL；URL binding 在内容被可信 resolver 物化并验证之前保持 readiness `Unknown`。不要把 secret 放进 inline text/object；密钥属于 credential Requirement。

## Implementation binding：从 role 到实现

Flow 只声明 capability 与 role：

```yaml
requires:
  capability: huoban.dev/designReview
  role: primary-review
```

Run 把 role 绑定到一个具体实现：

```yaml
bindings:
  primary-review:
    adapterRef: grill-me-skill-md
  spec-writer:
    skillRef: spec-outline-writer
  stage-reporter:
    adapterRef: stage-reporting-mcp
```

每个 binding 必须在 `skillRef` 与 `adapterRef` 中二选一。provider 必须声明 Flow 所需 capability；Preflight 会检查 role、对象引用与 capability 是否匹配。

### fallback

fallback 属于本次 Run binding，而不属于 Adapter：

```yaml
bindings:
  primary-review:
    adapterRef: preferred-reviewer
    fallback:
      skillRef: local-reviewer
```

fallback 不是运行时任意热切换。它必须显式锁定、提供同一 capability，并以与主 binding 相同的 requiredness 通过同一 Run 的 Preflight。当前 reference implementation 记录和检查 fallback 声明，但不提供自动切换 runtime。

## Snapshot 的准确边界

`snapshot` 整体可选；一旦存在，v1alpha1 要求以下五个 hash 全部出现：

| 字段 | 实际散列内容 |
| --- | --- |
| `flowSpecHash` | 当前 Flow 的 `spec`。 |
| `profileRefHash` | `Run.spec.profileRefs` 引用列表。 |
| `policyRefHash` | `Run.spec.policyRefs` 引用列表。 |
| `bindingHash` | `Run.spec.bindings`。 |
| `inputBindingHash` | `Run.spec.inputBindings`，包含 inline 值、内容引用与锁定 hash。 |

`profileRefHash` 与 `policyRefHash` 不等于被引用对象完整内容的 hash。若实现需要内容级不可变性，应使用 generation、外部内容寻址存储或未来标准字段；不能把引用列表 hash 描述成完整 Profile / Policy 快照。

Preflight Report 的 `snapshotHash` 又是另一层：它哈希完整 `Run.spec`，用来防止把旧报告套到已修改的 Run 上。它同样不归档 workspace 中所有对象内容。

## Preflight gate

Run scope Preflight 的摘要进入 `status`：

```yaml
status:
  conditions:
    - type: PreflightResolved
      status: "True"
      reason: DependenciesResolved
    - type: PreflightReady
      status: "True"
      reason: DependenciesReady
  preflight:
    scope:
      type: run
    snapshotHash: sha256:dc036d6f...
    checkedAt: 2026-07-15T00:00:00.000Z
    reportRef:
      kind: Artifact
      name: idea-to-spec-review-dry-run-preflight-report
```

execute 前必须同时满足：

1. scope 是 `run`，不是单个 stage。
2. report 的 snapshotHash 对应当前 `Run.spec`。
3. 报告未超过 `validUntil`（如果存在）。
4. `PreflightReady=True`。

这只记录机械就绪观察。Policy、Trust、Checkpoint 与实际执行权限仍需独立处理。详见 [Preflight 模型](/preflight-model)。

## Lifecycle invocation

Flow 声明 lifecycle handler，Run 用 bindings 锁定 provider；处理过程记录在 `status.lifecycleInvocations`：

```yaml
lifecycleInvocations:
  - handlerId: report-stage-success
    eventId: idea-to-spec-review-001-sharpen-idea-succeeded
    eventType: stage.succeeded
    stage: sharpen-idea
    phase: Pending
    attempts: 0
```

`eventId` 标识一个逻辑事件，重试应复用它。`attempts` 与 phase 记录投递观察状态，但当前 schema 和 reference CLI 不提供 dispatcher，也不承诺 exactly once。详见 [Lifecycle Handler](/lifecycle-handler-model)。

## Conditions、Checkpoint 与 Artifact

`Run.status.conditions` 是机器可读摘要，不替代专属对象：

| Run status | 专属对象 |
| --- | --- |
| `CheckpointRequired` | `Checkpoint` 保存 runRef、stage、reason、requiredActions 与 artifactRefs。 |
| `artifacts[]` | `Artifact` 保存类型、内容引用、hash、review 与可选 Run/stage 来源。 |
| `observedSideEffects[]` | `Policy` 定义长期允许、拒绝或审批规则。 |
| `lifecycleInvocations[]` | Flow 的 Lifecycle Handler 声明事件与 delivery。 |

不要把产物正文、审批评论或完整事件 payload 塞进 Run status。

## 结构示例

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Run
metadata:
  name: idea-to-spec-review-001
spec:
  flowRef:
    name: idea-to-spec-review
    generation: 2
  profileRefs:
    - name: huoban-open-source-project
      generation: 1
  policyRefs:
    - name: safe-defaults
  mode: dryRun
  bindings:
    primary-review:
      adapterRef: grill-me-skill-md
    spec-writer:
      skillRef: spec-outline-writer
    stage-reporter:
      adapterRef: stage-reporting-mcp
  inputBindings:
    idea:
      type: content
      contentRef:
        type: file
        path: examples/inputs/product-idea.md
      hash: sha256:2fa7baf3030dddd710b3818c589ff8378d0d0b7f44af28e60ab1e254d159ac60
  snapshot:
    flowSpecHash: sha256:35f574b2c801f40243ee2f6a1f96e2bba985c0fe93da887f1583a5fd1bd19269
    profileRefHash: sha256:d9ec168320ac8ea60d3aa5f049ce8a58bc0c625f1937d560c7e4b0d801f3d5d9
    policyRefHash: sha256:745db2bbc095810039b9b13ef544ce49bd28e1f7474757cf256e27906fb65006
    bindingHash: sha256:9d47cbf573e28dad7fd0080bdc96894a27ba85377421e4e34f3455cc58a5f01b
    inputBindingHash: sha256:44da7763a5145f0e4dccda5d8e66614372dce460fb65ef03f2c51d2544d33b8f
status:
  phase: WaitingForCheckpoint
  currentStage: sharpen-idea
  lifecycleInvocations:
    - handlerId: report-stage-success
      eventId: idea-to-spec-review-001-sharpen-idea-succeeded
      eventType: stage.succeeded
      stage: sharpen-idea
      phase: Pending
      attempts: 0
  artifacts:
    - name: review-notes
```

完整文件位于 source repository 的 `examples/runs/idea-to-spec-review-run.yaml`；生成路径和 Preflight 报告见 [Preflight 与 Lifecycle 示例](/example-preflight-lifecycle)。

## 常见错误

- 把 `profileRefHash` 写成 effective Profile 内容 hash。
- 同一个 binding 同时写 `skillRef` 与 `adapterRef`。
- 在 Run 启动后静默更换 provider 或 fallback。
- 给 Run 增加 Flow 未声明的输入，或让 binding 类型与入口类型不一致。
- 只保存文件 path 而不保存内容 hash，随后仍声称 Run 可重放。
- 用 stage scope Preflight 结果授权整次 Run。
- 把 `PreflightReady=True` 当作 Policy 已通过。
- 在 dry-run 中伪造已执行的 lifecycle invocation 或 Artifact。
- 把 Run 当成日志、队列或产物内容仓库。
