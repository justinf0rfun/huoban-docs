# Profile 对象

`Profile` 是 Huoban 的墨方：它不改变 Skill 本体，但改变 Skill 在某个项目、团队或场景里的判断标准。

规范页定义 Profile 的字段形状和验证边界。为什么上下文必须成为对象，请读 [Profile 模型](/profile-model)。

## 对象职责

Profile 解决的问题是：上下文不能只散落在聊天历史、AGENTS.md、CLAUDE.md、Cursor Rules 或个人记忆里。它必须能被分层、合并、导入、审查、快照和迁移。

Profile 负责描述：

- 团队规则。
- 项目结构。
- 架构边界。
- 领域语言。
- 验证命令。
- 历史决策。
- 输出风格。
- 风险提示。

Profile 不负责：

- 声明能力本身；这属于 `Skill` 或 `Adapter`。
- 执行权限控制；这属于 `Policy`。
- 编排阶段；这属于 `Flow`。
- 记录某次执行的实际上下文 hash；这属于 `Run.snapshot`。

## v1alpha1 字段

| 字段 | 要求 | 说明 |
| --- | --- | --- |
| `spec.layers` | 必填，至少 1 个 | 分层上下文数组。 |
| `spec.layers[].name` | 必填 | layer 名称，例如 `style`、`commands`、`architecture`、`risk`。 |
| `spec.layers[].data` | 条件必填 | 结构化上下文。`data` 或 `content` 至少存在一个。 |
| `spec.layers[].content` | 条件必填 | 自然语言上下文。`data` 或 `content` 至少存在一个。 |
| `spec.merge` | 可选，开放结构 | 合并策略。v1alpha1 不固定具体 DSL。 |
| `spec.policyRefs` | 可选 | 关联 Policy，但不内联权限规则。 |
| `status` | 可选 | import、merge、review 或 conflict 的状态。 |

`layers[].data` 允许任意结构化字段，因为不同项目的上下文形状不同。能进入公共标准的结构化字段，必须先在真实 workflow 中产生重复证据。

## 最小合法示例

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Profile
metadata:
  name: project-profile
spec:
  layers:
    - name: style
      data:
        preferSmallChanges: true
        avoidUnrelatedRefactor: true
      content: |
        Prefer small surgical changes. Do not refactor unrelated code.
    - name: commands
      data:
        testCommand: npm test
      content: |
        Run npm test before finalizing changes.
```

参考 example：`examples/profiles/project-profile.yaml`。

## 验证规则

validator 至少应检查：

- `kind` 必须是 `Profile`。
- `spec.layers` 至少包含一个 layer。
- 每个 layer 必须有 `name`。
- 每个 layer 必须至少有 `data` 或 `content`。
- `policyRefs` 只能引用本地对象形状；是否存在由 workspace-level validation 判断。
- 从 Markdown 导入的结构化字段如果是推断出来的，应写入 `ImportNeedsReview` condition。

## 合并建议

v1alpha1 schema 允许 `merge` 为开放结构，但实现者应保持可解释：

- 按 `profileRefs` 顺序应用。
- 同名 layer 可以合并，但冲突必须可见。
- scalar 覆盖、list append 或 replace 策略必须可解释。
- 生成 effective Profile 时应能计算 hash，并写入 Run snapshot。

这些是实现建议，不是新的 schema 字段承诺。

## 与相邻对象的边界

| 容易混淆 | 边界 |
| --- | --- |
| `Profile` / `Policy` | Profile 影响判断标准；Policy 控制副作用、审批和权限。 |
| `Profile` / `Skill` | Profile 是上下文；Skill 是能力。 |
| `Profile` / `Adapter` | 上下文-only 文件导入为 Profile；外部能力或工具流程导入为 Adapter。 |
| `Profile` / `Run` | Profile 是可复用上下文对象；Run 锁定某次执行实际使用的 Profile 引用和 hash。 |

## 常见错误

- 把 `publishRemote`、`modifyGitHistory` 这类权限规则只写在 Profile 自然语言里。应派生或手写 `Policy`。
- 把外部工具调用步骤写进 Profile。应使用 `Adapter` 或 `Flow`。
- 把团队私有实践写成公共标准字段。应先保留在 `layers[].data`。
- 合并 Profile 时静默覆盖冲突。应写入 condition 或让用户审查。

## 活字印刷位置

Profile 对应“墨方”。同一枚活字在不同墨色下会呈现不同效果；同一个 Skill 在不同 Profile 下会产生不同判断。Profile 让 Huoban 可润墨。
