# 示例：导入 AGENTS.md 为 Profile

这个示例展示如何把已有 `AGENTS.md` 从普通 Markdown 上下文升级为 Huoban `Profile`。

它验证的核心假设是：项目上下文不应该只靠 agent 在聊天里临时读取。上下文应该被分层、结构化、保留原文、标记推断，并能继续派生 Policy。

## 使用场景

适合从这些文件开始：

- `AGENTS.md`
- `CLAUDE.md`
- Cursor Rules
- 项目内的 AI 使用说明
- 团队工程规则文档

这些文件通常包含风格偏好、验证命令、风险边界和领域语言。它们不是能力本身，因此默认应进入 `Profile`，不是 `Skill` 或 `Adapter`。

## 示例文件

| 路径 | 角色 |
| --- | --- |
| `examples/import/AGENTS.md` | 原始输入。 |
| `examples/generated/agents-profile.yaml` | CLI 生成的 Profile。 |
| `examples/export/AGENTS.md` | 从 Profile 导出的 Markdown。 |
| `examples/derived/agents-risk-policy.yaml` | 从 risk layer 派生的 Policy。 |

## 对象关系

```text
examples/import/AGENTS.md
  -> Profile/agents-profile
      layers: style, commands, risk
      conditions: ProfileImported, ImportNeedsReview
  -> Policy/agents-profile-risk-policy
      rules: requireApproval for risky side effects
  -> examples/export/AGENTS.md
      reviewable Markdown generated from Profile
```

`Profile` 保留上下文和结构化线索；`Policy` 才负责副作用、审批和权限边界。这个拆分让上下文可润墨，让风险规则可审查。

## 运行命令

在 source repo 中运行：

```bash
npm run import:profile
npm run export:agents
npm run derive:policy
npm run validate
npm run explain
```

对应的底层命令形态是：

```bash
node ./bin/huoban.js import examples/import/AGENTS.md --as Profile --out examples/generated/agents-profile.yaml
node ./bin/huoban.js export examples/generated/agents-profile.yaml --to AGENTS.md --out examples/export/AGENTS.md
node ./bin/huoban.js derive policy examples/generated/agents-profile.yaml --out examples/derived/agents-risk-policy.yaml
```

## 输入片段

```md
# Agent Rules

## Style

Prefer small surgical changes. Do not refactor unrelated code.
Use test-first changes for bug fixes when practical.

## Commands

Run `npm run validate` before finalizing schema or example changes.
Use `npm run explain` to inspect the example workspace.

## Risk

Ask before modifying git history, publishing remote changes, or accessing secrets.
```

## 关键 Profile 片段

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Profile
metadata:
  name: agents-profile
  labels:
    huoban.dev/imported-from: examples/import/AGENTS.md
spec:
  layers:
    - name: style
      data:
        preferSmallChanges: true
        avoidUnrelatedRefactor: true
        testFirstForBugfixes: true
      content: |-
        Prefer small surgical changes. Do not refactor unrelated code.
        Use test-first changes for bug fixes when practical.
    - name: commands
      data:
        commands:
          - npm run validate
          - npm run explain
      content: |-
        Run `npm run validate` before finalizing schema or example changes.
        Use `npm run explain` to inspect the example workspace.
    - name: risk
      data:
        requiresApproval:
          - publishRemote
          - modifyGitHistory
          - accessSecret
        inferred: true
      content: Ask before modifying git history, publishing remote changes, or accessing secrets.
status:
  conditions:
    - type: ProfileImported
      status: "True"
      reason: ImportedFromMarkdown
    - type: ImportNeedsReview
      status: "True"
      reason: RuleBasedInference
```

完整文件：`examples/generated/agents-profile.yaml`。

## 字段如何理解

| 字段 | 说明 |
| --- | --- |
| `metadata.labels.huoban.dev/imported-from` | 记录来源，方便审查和追溯。 |
| `spec.layers[].name` | 把 Markdown 章节拆成稳定上下文层。 |
| `spec.layers[].content` | 保留原文，避免结构化推断丢失语义。 |
| `spec.layers[].data` | 提供 agent 和 tooling 可读取的结构化线索。 |
| `spec.policyRefs` | 本示例没有直接写入；风险规则先派生成独立 Policy，再由 Flow 或 Run 引用。 |
| `status.conditions[].ImportNeedsReview` | 明确告诉读者：导入结果包含推断，需要 review。 |

## explain 应该看什么

`npm run explain` 会把生成对象解释为：

```text
Profile/agents-profile
  layers: 3
    - style (data + content)
    - commands (data + content)
    - risk (data + content)
  conditions:
    - ProfileImported: True (ImportedFromMarkdown)
    - ImportNeedsReview: True (RuleBasedInference)
```

这说明导入没有只复制 Markdown，而是保留了三层上下文，并显式标出结构化推断。

## print / dry-run 如何使用它

这个页面本身不生成 Run。它提供的是 `Profile/agents-profile` 这种可润墨对象。

当某个 Flow 或 Run 引用这个 Profile 时，`print:dry-run` 应把 Profile 引用和有效内容 hash 锁进 `Run.spec.snapshot`。这时 Profile 不再只是 Markdown 背景材料，而是一次 AI 工作的可追溯上下文来源。

## 派生 Policy

`risk` layer 中的审批要求不应长期停留在 Profile 里。它可以派生为 Policy：

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Policy
metadata:
  name: agents-profile-risk-policy
spec:
  defaults:
    unknownSideEffect: requireApproval
  rules:
    - effect: requireApproval
      when:
        sideEffects:
          - publishRemote
          - modifyGitHistory
          - accessSecret
```

完整文件：`examples/derived/agents-risk-policy.yaml`。

## 在三句话中的位置

- **可排版**：导入后的 Profile 可以被 Flow 或 Run 引用，不再靠人工粘贴上下文。
- **可润墨**：style、commands、risk 等 layer 改变 agent 的判断环境，但不改变 Skill 本体。
- **可印刷**：Run snapshot 可以记录使用了哪套 Profile，审查时能复盘上下文来源。

## 常见修改方式

| 你要改什么 | 应该改哪里 |
| --- | --- |
| 增加团队编码风格 | 增加或修改 `Profile.spec.layers`。 |
| 增加必须运行的命令 | 修改 `commands` layer。 |
| 增加审批或权限规则 | 派生或维护 `Policy`，不要只写自然语言。 |
| 导出给现有 agent 工具使用 | 使用 `examples/export/AGENTS.md` 形态，但保留 review。 |

## 常见错误

| 错误 | 为什么错 |
| --- | --- |
| 把 AGENTS.md 导入为 Adapter。 | 这个文件主要是上下文，不是外部能力接口。 |
| 删除原文只保留结构化 data。 | 会丢失语义，agent 和人都无法复盘推断来源。 |
| 把 `requiresApproval` 当作 Profile 的最终权限模型。 | Profile 影响判断标准；Policy 才约束副作用和审批。 |
| 看到导入成功就直接信任。 | `ImportNeedsReview` 表示结构化字段需要人类确认。 |
