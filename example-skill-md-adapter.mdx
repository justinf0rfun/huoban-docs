# 示例：SKILL.md Adapter

这个示例展示如何用手写 `Adapter` 描述工具生态里的 `SKILL.md` 来源，但不直接把它升级为可信的 `Skill`。

它验证的核心假设是：已有 AI 工具能力可以被复用，但必须先补齐 capability、输入输出映射、副作用、checkpoint 和 trust 边界，才能进入可编排对象体系。

## 使用场景

适合从这些来源开始：

- Codex skill
- Claude skill
- 本地 `SKILL.md`
- MCP tool 描述
- Cursor Rule 中包装出的工具流程
- 团队内部已有的 prompt-driven 工具说明

如果来源主要描述“怎么执行一种能力”，可以建模为 `Adapter`。如果来源主要描述“项目规则和上下文”，应导入为 `Profile`。

当前 source repo 没有 `skills/grill-me/SKILL.md` fixture，CLI 也没有 `import --as Adapter`。因此本页不是可复现导入流程；它验证的是一个已手写的 Adapter YAML 如何被 schema 校验和 explain 解释。

## 示例文件

| 路径 | 角色 |
| --- | --- |
| `examples/adapters/grill-me-skill-md.yaml` | 手写 Adapter，描述一个外部 SKILL.md 来源。 |
| `examples/trust/local-reviewed-grill-me.yaml` | Adapter 来源的 Trust 记录。 |
| `examples/policies/safe-defaults.yaml` | 约束副作用的 Policy。 |
| `examples/flows/idea-to-spec-review.yaml` | 使用 capability 的 Flow。 |
| `examples/runs/idea-to-spec-review-run.yaml` | 把 Flow role 绑定到 Adapter 的 Run。 |

## 对象关系

```text
external SKILL.md source
  -> Adapter/grill-me-skill-md
      source: skillMd skills/grill-me/SKILL.md
      capabilities: huoban.dev/designReview, huoban.dev/planning
      mapping: user.prompt -> artifact.decisionNotes
      sideEffects: readFile, writeFile
      trustRef: local-reviewed-grill-me
  -> Trust/local-reviewed-grill-me
      level: reviewed
      sandbox: network deny
      policyRefs: safe-defaults
  -> Run binding
      primary-review -> Adapter/grill-me-skill-md
```

Adapter 是外部能力的标准化边界；Trust 记录来源是否被审查；Policy 控制副作用和审批。这里的 `skills/grill-me/SKILL.md` 是 Adapter 指向的外部来源路径，不是当前仓库内可复现 fixture。

## 关键 Adapter 片段

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Adapter
metadata:
  name: grill-me-skill-md
  labels:
    huoban.dev/imported-from: SKILL.md
spec:
  source:
    type: skillMd
    path: skills/grill-me/SKILL.md
  capabilities:
    - huoban.dev/designReview
    - huoban.dev/planning
  mapping:
    inputs:
      prompt: user.prompt
      context: profile.effectiveContent
    outputs:
      review: artifact.decisionNotes
  sideEffects:
    declared:
      - readFile
      - writeFile
  checkpoint:
    recommended: true
    reason: SkillProducesDesignJudgment
  trustRef:
    name: local-reviewed-grill-me
status:
  conditions:
    - type: AdapterImported
      status: "True"
      reason: ImportedFromSkillMd
    - type: ImportNeedsReview
      status: "True"
      reason: CapabilityAndSideEffectsInferred
```

完整文件：`examples/adapters/grill-me-skill-md.yaml`。

## 工具 skill 与 Huoban Skill

两者的共性是：都描述一类可复用 AI 能力。

关键差异在于边界：

| 维度 | 工具安装的 skill | Huoban `Skill` |
| --- | --- | --- |
| 主要用途 | 让某个工具知道如何执行。 | 声明可验证、可组合的能力对象。 |
| 输入输出 | 常隐藏在自然语言或工具约定里。 | 应显式声明 `inputs` 和 `outputs`。 |
| 副作用 | 可能未声明或只写在说明里。 | 必须声明 `sideEffects`。 |
| 信任 | 依赖安装来源或用户判断。 | 通过 `Trust`、`Policy` 和 review 状态表达。 |
| 编排 | 通常绑定在具体工具内。 | 可被 `Flow` capability-first 引用。 |

因此把 `SKILL.md` 纳入 Huoban 时，默认建模目标应是 `Adapter`，不是直接生成原生 `Skill`。只有当 capability、输入输出、副作用和 trust 都被审查后，才适合沉淀为 Huoban `Skill`。

## 中文活字的启发

中文活字不是只有“整字”这一层。偏旁、部首、笔画和字形结构共同决定一个字能否被识别、归类和复用。

工具 skill 也是这样：一个 `SKILL.md` 往往混合了能力描述、触发条件、上下文假设、工具限制和安全边界。Huoban 不把它整块吞进 Flow，而是用 Adapter 先拆出可声明部分：

- capability 像可识别的字形。
- mapping 像字形和版式之间的位置关系。
- sideEffects 像印刷时可能影响纸张、墨色或版面的副作用。
- trustRef 像这枚异形活字是否经过验收。

这让外部 skill 可以进入版盘，但不会污染 Huoban 的标准对象语义。

## Trust 记录

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Trust
metadata:
  name: local-reviewed-grill-me
spec:
  subjectRef:
    kind: Adapter
    name: grill-me-skill-md
  source:
    type: local
    path: skills/grill-me/SKILL.md
  level: reviewed
  review:
    reviewer: core-maintainer
    reviewedAt: "2026-06-23T12:00:00Z"
    conclusion: approved-for-local-use
  sandbox:
    required: true
    network: deny
  policyRefs:
    - name: safe-defaults
```

完整文件：`examples/trust/local-reviewed-grill-me.yaml`。

Trust 不让 Adapter 自动变安全。它记录来源、review 结论、sandbox 要求和关联 Policy，让下游 runner 或 registry tooling 能做保守决策。

## 在 Flow 中如何使用

Flow 只声明 capability：

```yaml
stages:
  - id: sharpen-idea
    requires:
      capability: huoban.dev/designReview
      role: primary-review
```

Run 再绑定具体 Adapter：

```yaml
bindings:
  primary-review:
    adapterRef: grill-me-skill-md
```

这说明 `Adapter/grill-me-skill-md` 是本次执行的实现选择，不是 Flow 的永久依赖。

## 运行验证

在 source repo 中运行：

```bash
npm run validate
npm run explain
```

`npm run explain` 应能看到：

```text
Adapter/grill-me-skill-md
  source: skillMd skills/grill-me/SKILL.md
  capabilities: huoban.dev/designReview, huoban.dev/planning
  declaredSideEffects: readFile, writeFile
  trustRef: local-reviewed-grill-me
```

这说明该 Adapter 已经具备 capability、来源、副作用和 trust 引用。

注意：这里验证的是静态 Adapter 对象，不验证从 `SKILL.md` 自动生成 Adapter。要把本页升级为迁移型示例，需要新增真实 `SKILL.md` fixture 和 Adapter import CLI。

## print / dry-run 如何使用它

这个页面本身不直接生成 Run。它提供的是 `Adapter/grill-me-skill-md`，供 Run binding 使用。

在 `idea-to-spec-review` 中，`print:dry-run` 会把 Flow 的 `primary-review` role 绑定到这个 Adapter：

```yaml
bindings:
  primary-review:
    adapterRef: grill-me-skill-md
```

生成的 dry-run Run 不执行这个 Adapter，但会把 binding 写入锁定快照。审查者因此能看到：这次计划使用的是外部 `SKILL.md` 来源，而不是原生 Huoban Skill。

## 在三句话中的位置

- **可排版**：Adapter 声明外部来源能满足哪些 capability，让 Flow 可以按 capability 排版。
- **可润墨**：Adapter 可以读取 `profile.effectiveContent`，但不把 Profile 内容混进外部 skill。
- **可印刷**：Run binding 和 Trust 记录让一次使用外部 skill 的计划可审查、可复盘。

## 常见修改方式

| 你要改什么 | 应该改哪里 |
| --- | --- |
| 支持新的外部 skill 来源 | 新增 `Adapter.spec.source` 和 mapping。 |
| 调整能力声明 | 修改 `Adapter.spec.capabilities`，并重新 review。 |
| 增加网络、写文件、发布等风险 | 修改 `sideEffects`，并让 Policy 约束。 |
| 标记来源可信度 | 新增或修改 `Trust`。 |
| 在某次执行中使用 Adapter | 修改 `Run.spec.bindings`。 |

## 常见错误

| 错误 | 为什么错 |
| --- | --- |
| 导入 SKILL.md 后直接生成 trusted Skill。 | 外部来源通常缺少标准输入输出、副作用和 Trust 证据。 |
| 把 Profile 上下文写进 Adapter。 | Adapter 描述外部能力，Profile 描述上下文。 |
| 只声明 capability，不声明 side effects。 | runner 无法判断是否需要审批或 sandbox。 |
| 认为 Trust 替代 Policy。 | Trust 记录来源信任；Policy 控制副作用和权限。 |
