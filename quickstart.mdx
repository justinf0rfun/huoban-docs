Huoban 是 AI 系统的开放编排协议：可排版、可润墨、可印刷。

本页复现当前公开实现的完整准备路径：

```text
validate -> explain / print --dry-run -> preflight -> Policy / Trust / Checkpoint -> execute
```

当前仓库实现到 Preflight 证据生成。它没有 `execute` 命令、agent runtime 或 Lifecycle dispatcher；最后一步由未来或第三方 runtime 实现。

<Warning>
  dry-run 不是 Preflight，Preflight 也不是授权。`execute` Run 必须具有 Run scope 的 `PreflightReady=True`，之后仍要独立应用 Policy、Trust、Checkpoint、sandbox 和平台权限。
</Warning>

## 前置条件

- Git。
- Node.js 20 或更高版本。
- 一个可以写入本地工作区的 shell。

文中的 `examples/...` 是 Huoban source repo 内的文件路径，不是本文档站的网页链接。文档站读者不一定能直接浏览该工作区，因此示例页会内嵌关键 YAML。

<Steps>
  <Step title="获取源码并安装依赖">

```bash
git clone git@github.com:justinf0rfun/huoban.git
cd huoban
npm install
```

确认 Node.js 版本：

```bash
node --version
```

  </Step>

  <Step title="验证 10 个顶层 kind">

```bash
npm run validate
```

当前 example workspace 包含 18 个对象，覆盖 10 个顶层 kind：`Skill`、`Profile`、`Flow`、`Run`、`Checkpoint`、`Policy`、`Adapter`、`Artifact`、`Registry`、`Trust`。

验证单个对象：

```bash
npm exec -- huoban validate examples/flows/idea-to-spec-review.yaml
```

预期输出：

```text
✓ examples/flows/idea-to-spec-review.yaml: Flow/idea-to-spec-review
```

`Requirement`、Preflight 和 Lifecycle Handler 是嵌入式执行契约，不是额外的 kind。

  </Step>

  <Step title="解释 workspace">

```bash
npm run explain
```

预期输出开头：

```text
Huoban Workspace
  objects: 18
  kinds: Adapter=2, Artifact=2, Checkpoint=1, Flow=1, Policy=2, Profile=3, Registry=1, Run=2, Skill=2, Trust=2
  print process:
```

只解释参考 Flow：

```bash
npm run explain:flow
```

`explain` 读取对象并显示结构、引用和条件，不创建 Run，也不检查目标环境是否可用。

  </Step>

  <Step title="打印 dry-run Run">

```bash
npm run print:dry-run
```

该命令把 `Flow/idea-to-spec-review` 打印为 `Run/idea-to-spec-review-dry-run`，不会执行 stage 或 handler。脚本把 `examples/inputs/product-idea.md` 绑定到 Flow 声明的 required `idea: file` 入口：

```bash
npm exec -- huoban print --dry-run examples/flows/idea-to-spec-review.yaml \
  --out examples/generated/idea-to-spec-review-dry-run.yaml \
  --input idea=file:examples/inputs/product-idea.md \
  --bind primary-review=adapter:grill-me-skill-md \
  --bind spec-writer=skill:spec-outline-writer \
  --bind stage-reporter=adapter:stage-reporting-mcp
```

生成的 `Run.spec.inputBindings.idea` 保存文件内容引用与 raw-byte hash。`Run.spec.snapshot` 使用五个固定字段：

| 字段 | 当前 CLI 哈希的内容 |
| --- | --- |
| `flowSpecHash` | `Flow.spec`。 |
| `profileRefHash` | 本次 `profileRefs` 数组，不是 Profile 内容。 |
| `policyRefHash` | 本次 `policyRefs` 数组。 |
| `bindingHash` | 本次 `bindings` 对象，包括写在 binding 内的 fallback。 |
| `inputBindingHash` | 本次 `inputBindings`，包括内容引用、inline 值和文件 hash。 |

预期输出：

```text
✓ examples/flows/idea-to-spec-review.yaml -> examples/generated/idea-to-spec-review-dry-run.yaml: Run/idea-to-spec-review-dry-run
```

`mode: dryRun` 只表示没有执行 agent 工作，不表示依赖已经 ready。

  </Step>

  <Step title="执行默认离线 Preflight">

```bash
npm run preflight
```

该脚本更新 dry-run Run，并生成 `Artifact/idea-to-spec-review-dry-run-preflight-report`。Preflight report 的 `snapshotHash` 哈希整个 `Run.spec`，与上面的五个分项 hash 不同。

默认行为是 offline。离线模式会检查入口声明、stage 来源、input binding、文件内容 hash、对象引用、generation、capability、implementation binding、本地路径和 executable；远程 context、MCP 与不可取得的远程 Artifact 内容 readiness 保持 `Unknown`。只有明确需要动态 discovery 时才使用 `--live`。

查看 CLI 参数：

```bash
npm exec -- huoban --help
```

`--live` 只执行有界、只读 discovery。它不会调用 MCP tool。

  </Step>

  <Step title="检查执行前安全边界">

查看这些 source tree 文件：

```text
examples/policies/safe-defaults.yaml
examples/trust/local-reviewed-grill-me.yaml
examples/trust/local-stage-reporting.yaml
examples/checkpoints/idea-to-spec-review-direction.yaml
examples/generated/idea-to-spec-review-dry-run.yaml
examples/generated/idea-to-spec-review-preflight-report.yaml
```

Run scope 的 `PreflightReady=True` 只说明该 Run snapshot 的 required dependencies 在记录时刻 ready。它不会批准副作用，不会提升 Trust，不会解决 Checkpoint，也不会启动执行。

  </Step>
</Steps>

## 一次运行完整检查

完成上面的逐步检查后，可以运行仓库定义的完整测试：

```bash
npm test
```

它会运行 unit tests、Profile import、Policy derive、AGENTS.md export、dry-run print、Preflight、schema validation、workspace explain 和 generated-file drift check。该命令会重写仓库中的 deterministic generated examples；只想只读检查时使用 `npm run validate` 和 `npm run explain`。

## 如何判断结果

| 结果 | 可以确认什么 | 不能据此确认什么 |
| --- | --- | --- |
| `validate` 通过 | 对象符合当前 JSON Schema。 | 引用存在、依赖 ready 或执行获准。 |
| `explain` 输出连贯 | CLI 能解释对象关系。 | 已创建执行快照或完成环境探测。 |
| dry-run Run 已生成 | Flow、refs、动态输入、implementation bindings 和 hashes 已被打印。 | 依赖 ready、Policy 允许或 stage 已运行。 |
| Run scope `PreflightReady=True` | required dependencies 对该 `Run.spec` snapshot ready。 | Policy、Trust、Checkpoint、sandbox 或平台授权已通过。 |
| stage scope `StagePreflightReady=True` | 指定 stage 的 required checks ready。 | 整个 Run 可以执行。 |
