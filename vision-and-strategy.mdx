Huoban 是 AI 系统的开放编排协议：**可排版、可润墨、可印刷**。

它定义 AI 原生工作的共享对象语言，让能力、上下文、编排、执行、校对、治理、适配、产物、索引与信任可以被不同 agent、tool、MCP server 和 runner 共同理解。

Huoban 的顺序是 **standard-first、runtime-later**。项目先验证对象语义、schema、examples、解释、dry-run 与 Preflight 可以跨工具保持一致，再扩展执行 runtime。

## 项目定位

Huoban 不是另一套私有 workflow，也不是完整 agent runtime。它解决的是描述层的互操作问题：

- 能力怎样被命名、组合和替换？
- 上下文怎样被分层、审查和迁移？
- 一类工作怎样形成可复用版式？
- 一次工作怎样锁定 refs 与 bindings？
- 执行前怎样检查并记录依赖就绪状态？
- 哪些行为必须受 Policy、Trust 或 Checkpoint 约束？
- 产物与生命周期观察怎样留下可复盘事实？

这使不同执行环境可以实现同一组协议对象，而不要求使用同一个模型、IDE、MCP server 或 runner。

## 第一批用户

Huoban 首先服务已经积累 AI 工作方法、但缺少跨工具标准化的人：

- 在多个 agent 或 IDE 间迁移规则与 skill 的开发者。
- 需要组合多个 Skill、MCP tool 和上下文来源的团队。
- 需要在真实公司项目中审查副作用、依赖和执行证据的平台工程团队。
- 构建编排系统、希望采用开放对象边界而不是自造私有 DSL 的实现者。

他们的核心问题不是“如何调用模型”，而是资产分散、上下文隐式、依赖运行时才失败、工作流无法复盘，以及治理语义绑定在单一工具里。

## 第一个不可替代价值

Huoban 的第一价值不是执行一个 Flow，而是把已经存在于规则文件、skill、MCP、prompt、聊天习惯和内部系统中的 AI 工作方法，变成可命名、可验证、可组合、可审查、可迁移的对象与共享结构。

因此当前优先级是：

1. 对象语义与边界。
2. JSON Schema 与真实 examples。
3. validate、explain、import/export、dry-run。
4. Requirement 与 Preflight。
5. Policy、Trust、Checkpoint 和 Artifact 证据链。
6. 多实现 conformance。
7. 完整 runtime 后置。

## 十个顶层对象

v1alpha1 定义十个 `kind`：

| 层次 | 对象 | 职责 |
| --- | --- | --- |
| 能力 | `Skill` | 声明可复用 capability、输入输出、Requirement 与副作用。 |
| 上下文 | `Profile` | 声明分层上下文、外部上下文依赖与 Policy 引用。 |
| 编排 | `Flow` | 声明 capability-first stage、checkpoint、outputs 与 lifecycle。 |
| 执行 | `Run` | 锁定一次工作的 refs、mode、bindings 与 snapshot 字段。 |
| 决策 | `Checkpoint` | 声明一次 Run/stage 上的显式判断边界。 |
| 治理 | `Policy` | 声明 allow、deny、requireApproval 与未知副作用默认规则。 |
| 适配 | `Adapter` | 把外部 skill、MCP tool、规则文件或工具能力映射进协议。 |
| 产物 | `Artifact` | 记录内容引用、hash、review 与可选 Run/stage 来源。 |
| 发现 | `Registry` | 声明对象来源与 kind 索引范围。 |
| 信任 | `Trust` | 记录来源、review、level、签名、sandbox 与 Policy 引用。 |

文化隐喻进入解释层，API 使用全球开发者熟悉的英文对象名。不要在 schema 中创建拼音对象名。

## 共享执行契约

三个问题不需要膨胀成新 kind：

| 契约 | 位置 | 解决的问题 |
| --- | --- | --- |
| `Requirement` | Skill/Profile/Adapter 的 `requirements[]` | 一个对象依赖哪些文件、程序、MCP、凭据、runner capability 或上下文来源？ |
| `Preflight` | Run conditions/status + Preflight Report Artifact | 当前 Run 在目标 workspace 中是否可解析、可用？ |
| `Lifecycle Handler` | `Flow.spec.lifecycle[]` + Run bindings/status | 哪些有限事件需要上报、观察或清理，投递状态如何审计？ |

它们分别处理依赖声明、机械就绪和生命周期观察。它们不替代 Policy、Trust、Checkpoint，也不引入任意 hook 控制流。

## Capability-first 与 binding

Flow 先声明能力和 role：

```yaml
stages:
  - id: review
    requires:
      capability: huoban.dev/designReview
      role: primary-review
```

Run 再绑定具体实现：

```yaml
bindings:
  primary-review:
    adapterRef: grill-me-skill-md
```

这样 Flow 不被某个 skill 或 MCP server 永久锁定。Registry 可以帮助发现候选对象，但不替 Run 选择实现；Preflight 验证 binding 与 capability，不执行 provider。

## 完整执行前链路

<Steps>
  <Step title="Validate">
    验证对象字段与 fragment 形状。
  </Step>
  <Step title="Explain / dry-run">
    展开关系并生成计划态 Run；动态就绪可以保留 Unknown。
  </Step>
  <Step title="Preflight">
    在 Run scope 解析 refs、bindings 与 Requirement，保存逐项证据。
  </Step>
  <Step title="Policy / Trust / Checkpoint">
    独立处理副作用、来源信任、sandbox 与显式决策。
  </Step>
  <Step title="Execute">
    实现只能在 Run scope `PreflightReady=True` 且治理门满足后启动。
  </Step>
</Steps>

dry-run 不是 Preflight；Preflight 不是授权。把三者合并成一个“可以运行”状态会让失败原因和责任边界不可审计。

## 当前 reference implementation

当前仓库提供：

- 十种对象与共享 fragment 的 v1alpha1 schema。
- 覆盖对象链路的 YAML examples。
- `validate`、`explain`、import/export 与 `print --dry-run`。
- Run/stage scope Preflight、默认 offline，以及显式 `--live` 的只读 context source `HEAD` 与有限 MCP discovery。
- Preflight Report Artifact、Run conditions/status 与 lifecycle invocation 结构。
- Lifecycle Handler 声明、有限事件枚举与事件信封示例。

当前**不**提供：

- 完整 stage 执行 runtime。
- Lifecycle dispatcher 或实际 MCP tool 调用。
- Policy/Trust evaluator 和审批系统。
- hosted Registry、marketplace 或远程对象分发。
- 自动依赖安装、credential 获取或 fallback 热切换。

文档不得把 schema、Preflight 或示例写成这些 runtime 能力已经完成。

## 兼容与版本策略

当前 `huoban.dev/v1alpha1` 仍处于同一 alpha 草案周期，尚未承诺兼容稳定性。Requirement、Preflight 和 Lifecycle 作为本周期执行契约的一部分直接纳入 v1alpha1。

这次选择意味着：

- 新 validator 必须继续接受此前合法且未被明确收紧的 v1alpha1 对象。
- 旧 validator 会因 `additionalProperties: false` 拒绝新字段，不能声称旧 reader 前向兼容。
- 在发布稳定版本前，任何字段变化都必须同步 schema、CLI、tests、examples、source docs、Mintlify 与 changelog。
- 进入 beta/stable 后，破坏性变更必须升级 API 版本并提供迁移说明。

## 公司实践如何反馈标准

公共标准与公司定制必须分开。公司项目可以用私有 Profile、Policy、Adapter、Flow 和 Run 验证真实需求，再把重复证据反馈到公共标准。

一个概念进入公共标准前，应满足：

1. 至少两个独立真实 workflow 反复出现。
2. 不依赖某家公司、单一 agent 或隐藏 prompt 行为。
3. 能被 schema 验证、CLI 解释并由 example 展示。
4. 提升跨实现互操作性，而不是只方便一个 runtime。
5. 安全、兼容和失败语义已经明确。

未满足时，应留在私有对象、Adapter、example 或实现扩展里。

## 开源协作与治理

早期治理应保持决策速度，但所有标准变化必须留下公开、可审查证据：

- schema 和 fragment 变更。
- 至少一个真实 example。
- 行为测试与生成产物。
- 规范、模型、迁移和安全文档。
- changelog 与兼容性说明。
- 对抗审查记录或等价 review。

随着实现者和贡献者增加，项目可以建立明确的 maintainer 与 spec review 机制。中立治理是长期方向，不应在缺少真实生态时先设计庞大组织结构。

## 成功标准

Huoban 的成功不由“支持多少 agent”单独衡量，而由互操作事实衡量：

- 同一 Flow 能被不同实现解释，role/capability 语义一致。
- 同一 Profile 能跨工具迁移，内容与来源依赖可审查。
- 同一 Run 能解释 refs、bindings、snapshot 与 Preflight 证据。
- 不同实现对缺失依赖、未知 readiness、Policy 与 Checkpoint 给出一致边界。
- 公开 examples 能在多个实现中得到语义等价结果。

## 非目标

当前阶段不追求：

- 在对象模型验证前完成通用 runner。
- 把任意 shell hook 标准化为 Lifecycle Handler。
- 把所有 prompt、tool、resource 或 permission 升级为新 kind。
- 在 Trust 和分发语义稳定前建设 marketplace。
- 用视觉编排器掩盖未稳定的文本协议。
- 把某家公司的一次实践直接写成行业标准。

Huoban 的第一责任是把对象边界、执行契约和证据链定义准确。
