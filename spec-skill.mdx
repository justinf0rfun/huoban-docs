# Skill 对象

`Skill` 是 Huoban 的活字：一枚可被识别、复用、移动和重新排版的能力单元。

规范页只定义 Skill 的字段形状和验证边界。为什么 Skill 要独立于 Profile 和 Flow，请读 [Skill 是活字](/design-philosophy#skill-是活字) 与 [对象模型](/object-model)。

## 对象职责

Skill 解决的问题是：把“某个 AI 能力能做什么”从 prompt、工具 UI 或个人习惯中提取出来，变成可验证对象。

Skill 负责声明：

- 它提供哪些 `capabilities`。
- 它接受哪些输入、产生哪些输出。
- 它可能需要哪些 Profile layer。
- 它可能产生哪些副作用。
- 它是否建议或要求 checkpoint。
- 它是否来自某个 Adapter。

Skill 不负责：

- 保存项目上下文；这属于 `Profile`。
- 排列多个能力；这属于 `Flow`。
- 记录一次执行；这属于 `Run`。
- 批准危险操作；这属于 `Policy` 和 `Checkpoint`。
- 记录产物正文；这属于 `Artifact`。

## v1alpha1 字段

| 字段 | 要求 | 说明 |
| --- | --- | --- |
| `spec.capabilities` | 必填，至少 1 个，唯一 | capability 名称，格式为 `domain/name`，例如 `huoban.dev/designReview`。 |
| `spec.inputs` | 可选 | 输入声明。每项必须有 `name`，可附加实现侧字段。 |
| `spec.outputs` | 可选 | 输出声明。每项必须有 `name`，可声明 `artifactType`。 |
| `spec.profileHints` | 可选 | 推荐注入的 Profile layer 名称。 |
| `spec.sideEffects` | 必填 | 副作用声明对象。 |
| `spec.sideEffects.declared` | 必填 | 可能发生的标准 side effect 数组。 |
| `spec.checkpoint` | 可选，开放结构 | 建议或要求校对的提示，不是 Checkpoint 实例。 |
| `spec.adapterRef` | 可选 | 如果 Skill 由外部来源提升而来，引用来源 Adapter。 |
| `status` | 可选 | validator、registry 或 runtime 写入的事实状态。 |

`inputs`、`outputs` 和 `checkpoint` 允许实现侧附加字段；`capabilities` 和 `sideEffects.declared` 是跨工具互操作的核心字段。

## 最小合法示例

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Skill
metadata:
  name: design-review
spec:
  capabilities:
    - huoban.dev/designReview
  inputs:
    - name: proposal
      type: markdown
  outputs:
    - name: review
      artifactType: decision-notes
  sideEffects:
    declared:
      - readFile
      - writeFile
  checkpoint:
    required: true
    reason: human-design-decision
```

参考 example：`examples/skills/design-review.yaml`。

## 验证规则

validator 至少应检查：

- `kind` 必须是 `Skill`。
- `metadata.name` 必须是合法本地对象名。
- `spec.capabilities` 至少包含一个合法 capability，且不能重复。
- `spec.sideEffects.declared` 必须存在，即使数组为空。
- `sideEffects.declared` 只能使用标准枚举：`readFile`、`writeFile`、`deleteFile`、`executeCommand`、`modifyGitIndex`、`modifyGitHistory`、`networkAccess`、`sendMessage`、`publishRemote`、`spendMoney`、`accessSecret`、`installDependency`、`changePermission`。
- 如果存在 `adapterRef`，它必须能解析到 `Adapter` 或在 condition 中标记 unresolved。

## 与相邻对象的边界

| 容易混淆 | 边界 |
| --- | --- |
| `Skill` / `Adapter` | Skill 是原生能力声明；Adapter 是外部来源进入 Huoban 的标准化边界。 |
| `Skill` / `Profile` | Skill 不携带项目上下文；Profile 注入项目规则、领域语言和历史决策。 |
| `Skill` / `Flow` | Skill 声明能力；Flow 决定能力如何排列。 |
| `Skill` / `Policy` | Skill 声明可能副作用；Policy 决定是否允许、拒绝或要求审批。 |

## 常见错误

- 把团队代码风格写进 Skill。应写入 `Profile.layers`。
- 把执行顺序写进 Skill。应写入 `Flow.stages`。
- 省略 `sideEffects.declared`。没有副作用也要显式写 `declared: []`。
- 把外部 `SKILL.md` 直接视为 trusted Skill。默认应先生成 `Adapter`，review 后再提升。

## 活字印刷位置

Skill 对应“活字”。一枚活字必须有规格，才能进入字架、被选择、被排版、被润墨、被校对并最终参与印刷。Skill 的规格就是 capability、输入、输出和副作用边界。
