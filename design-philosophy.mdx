Huoban 活板的设计哲学来自活字印刷术。

活字印刷术的突破不在于“有字”。字早就存在。真正的突破在于：字可以被标准化、收纳、选择、排布、润墨、试印、校对、印刷、拆版、复用。

“活板”是这个隐喻的核心：不是把 AI 能力固定成一整块雕版，而是把 skill、context、flow、checkpoint、policy 和 artifact 做成可以拆开、重排、润墨、校对、复印的开放编排结构。

Huoban 是 AI 系统的开放编排协议：可排版、可润墨、可印刷。

## 核心判断

AI 原生工作不应该只停留在 prompt、聊天记录或某个单一工具里。它应该被拆成一组清晰的标准对象：

- 能力要像活字一样可选择、可组合、可复用。
- 上下文要像墨方一样可分层、可替换、可注入。
- 编排要像版式一样显式描述顺序、依赖、停顿和产物。
- 执行要像锁版印刷一样形成一次可追溯快照。
- 风险判断要像校对一样成为一等对象，而不是隐藏在临时对话里。

Huoban 的目标，是让 AI 工作变得明确、可编程、可检查、可复用、可分享。

## 对象映射

| Huoban 对象 | 活字印刷对应物 | 设计含义 |
| --- | --- | --- |
| `Skill` | 活字 | 原子能力，可被不同 Flow 反复选择和重排。 |
| `Profile` | 墨方 | 团队、项目、架构、领域、风格、历史和风险上下文。 |
| `Flow` | 版式 | 能力之间的顺序、输入输出、停顿和产物。 |
| `Run` | 锁版印刷 | 冻结 Flow、Profile、Policy 和 binding 的一次执行快照。 |
| `Checkpoint` | 校对 | 高判断力或高风险节点的人工或系统决策边界。 |
| `Policy` | 印坊规矩 | 自动化边界、审批规则和危险动作限制。 |
| `Adapter` | 异形活字转接件 | 把外部能力或导入格式标准化成可编排对象。 |
| `Artifact` | 印张 | 代码 diff、文档、报告、决策记录等产物。 |
| `Registry` | 字架 | 组织、索引和发现可用对象。 |
| `Trust` | 审校记录 | 记录来源、审查、信任等级和 sandbox 要求。 |

## 排印流程

<Steps>
  <Step title="取字：选择能力">
    从 Registry 中选择满足 capability 的 `Skill`。Skill 应该原子、独立、可复用，并显式声明输入、输出和副作用。
  </Step>
  <Step title="排版：组织 Flow">
    用 `Flow` 安排 stage、capability、artifact 和 checkpoint。Flow 描述的是工作如何被编排，不是某个工具的按钮顺序。
  </Step>
  <Step title="润墨：注入 Profile">
    用 `Profile` 注入团队规范、项目结构、领域语言、历史决策和输出要求。Profile 改变判断标准，不改变 Skill 本身。
  </Step>
  <Step title="立规：套用 Policy">
    用 `Policy` 声明副作用、审批、权限和危险动作边界。能自动执行的动作和必须人工确认的动作要分开。
  </Step>
  <Step title="锁版：生成 Run">
    用 `Run` 锁定一次执行使用的 Flow、Profile、Policy、binding 和输入。Run 是之后追溯、复盘和比较的事实来源。
  </Step>
  <Step title="校对：进入 Checkpoint">
    在高判断力或高风险节点进入 `Checkpoint`。人或系统可以 approve、request changes 或 reject。
  </Step>
  <Step title="付印：产出 Artifact">
    把代码 diff、文档、报告、决策记录或发布结果保存为 `Artifact`，让结果可以被审查、复用和分享。
  </Step>
</Steps>

## Skill 是活字

一个好的 `Skill` 应该像一枚规整的活字：

- 能被单独识别。
- 能被放进不同版式。
- 不偷偷携带某个项目的上下文。
- 不隐藏输入、输出和副作用。
- 能说明自己是否会读文件、写文件、执行命令、访问网络、花钱、发布远端变更或访问密钥。

它和今天很多 AI 工具里“安装的 skill”有直接关系，但不是同一个层级的概念。

共同点在于：二者都把一类可复用能力从临时 prompt 中抽出来，给它命名、保存，并让 agent 在合适的任务中调用。一个代码审查 skill、文档改写 skill、产品方案 grill skill，本质上都在解决“能力不要每次重新发明”的问题。

差异在于：工具里的 skill 通常是某个 agent runtime 可读取的操作说明，而 Huoban 的 `Skill` 是协议对象。它不仅要说明“怎么做”，还要声明 capability、输入、输出、副作用、上下文需求、信任信息，以及它能否被放进不同 `Flow` 中重排。

这里也可以借用汉字的部件结构来理解。传统活字通常以整字为字模，不是每次用偏旁部首现场拼字；但中文读者天然能理解，一个字并不是无结构的黑块，它可以被拆成偏旁、部首、笔画和构形关系。Huoban 的 `Skill` 也不是一整块不可分析的能力包，而是由 capability、inputs、outputs、side effects、trust metadata 等结构部件组成。只有这些部件清楚，`Skill` 才能被检索、校验、替换、组合和治理。

| 维度 | AI 工具安装的 skill | Huoban `Skill` |
| --- | --- | --- |
| 所属层级 | 某个工具或 agent runtime 的能力包 | 跨工具的标准对象 |
| 主要内容 | 操作说明、触发条件、参考资料 | capability、inputs、outputs、side effects、trust metadata |
| 结构粒度 | 往往是一整段可执行说明 | 可拆成标准字段并被机器验证 |
| 上下文处理 | 常和工具、项目或聊天历史耦合 | 项目上下文应放进 `Profile` |
| 编排方式 | 由工具决定何时调用 | 可被 `Flow` 显式排版 |
| 安全边界 | 依赖工具自己的权限模型 | 由 `Policy`、`Run`、`Checkpoint` 和 `Trust` 一起约束 |
| 迁移路径 | 直接安装到某个工具 | 先通过 `Adapter` 标准化，再视情况提升为原生 `Skill` |

因此，Huoban 不要求重写已有 skill 生态。更合理的路径是：把现有 `SKILL.md`、Claude Code skill、Codex skill 或其他工具能力先导入为 `Adapter`，补齐 capability、输入输出、副作用和信任信息，再决定它是否可以成为标准 `Skill`。详见 [导入 SKILL.md](/import-skill-md) 和 [Adapter 对象](/spec-adapter)。

示例：

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Skill
metadata:
  name: design-review
spec:
  capabilities:
    - huoban.dev/designReview
  inputs:
    - name: proposal
      type: markdown
  outputs:
    - name: review
      artifactType: decision-notes
  sideEffects:
    declared:
      - readFile
      - writeFile
```

## Flow 是版式

`Flow` 不只是 stage 列表。它描述一次工作应该如何排版：

- 这次任务需要哪些 capability？
- 哪个 stage 先运行，哪个后运行？
- 上一步产物如何传给下一步？
- 哪些位置必须 checkpoint？
- 使用哪一组 Profile 和 Policy？

示例：

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Flow
metadata:
  name: idea-to-spec-review
spec:
  profileRefs:
    - name: huoban-open-source-project
  policyRefs:
    - name: safe-defaults
  stages:
    - id: sharpen-idea
      requires:
        capability: huoban.dev/designReview
        role: primary-review
  checkpoints:
    - after: sharpen-idea
      type: humanApproval
      required: true
      reason: ConfirmDirectionBeforeSpec
```

## Profile 是润墨

同一枚活字，用不同墨色印出来，字形没变，但气质和用途变了。同一个 Skill，在不同上下文里运行，能力没变，但判断标准变了。

这就是 `Profile` 的位置。项目语言、架构边界、领域术语、历史决策、测试习惯和风险规则，都应该作为运行时上下文显式注入，而不是散落在临时 prompt 或聊天历史里。

## Checkpoint 是校对

自动化必须有边界。`Checkpoint` 让关键判断点显式出现：

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Checkpoint
metadata:
  name: idea-to-spec-review-001-direction
spec:
  runRef:
    name: idea-to-spec-review-001
  stage: sharpen-idea
  reason: ConfirmDirectionBeforeSpec
  requiredActions:
    - approve
    - requestChanges
    - reject
```

## 设计边界

  </Card>
</CardGroup>

## 设计原则

1. 标准化活字，不标准化文章。
2. Skill 尽量无状态。
3. Profile 是运行时上下文层。
4. Flow 描述能力编排，不绑定单一执行环境。
5. Run 必须可追溯。
6. 副作用必须可见。
7. 校对是一等能力。
8. 优先适配现有能力，不急着替代已有生态。
9. Flow 也应该可分享。
10. 文化隐喻必须进入结构，而不是停留在命名和文案上。
