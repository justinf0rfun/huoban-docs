`Flow` 是 Huoban 的版式：它声明 stage 如何排列、每个 stage 需要什么 capability、何处校对，以及哪些 lifecycle 集成跟随有限事件。Flow 是声明，不是 executor。

设计背景见 [Flow 模型](/flow-model)，Lifecycle fragment 见 [Lifecycle Handler](/spec-lifecycle-handler)。

## 顶层字段

| 字段 | 要求 | Schema 语义 |
| --- | --- | --- |
| `spec.profileRefs` | 可选 | local object reference 数组。 |
| `spec.policyRefs` | 可选 | local object reference 数组。 |
| `spec.stages` | 必填，至少 1 项 | stage 数组。 |
| `spec.checkpoints` | 可选 | Flow 内的 checkpoint 声明数组。 |
| `spec.lifecycle` | 可选 | Lifecycle Handler 数组。 |
| `status` | 可选 | 通用状态。 |

## Stage 字段

| 字段 | 要求 | Schema 语义 |
| --- | --- | --- |
| `id` | 必填 | 字符串；schema 不要求非空或唯一。 |
| `requires` | 与 `uses` 至少存在一个 | capability-first 需求对象。 |
| `requires.capability` | `requires` 内必填 | 合法 capability。 |
| `requires.role` | 可选 | Run binding key 字符串。 |
| `uses` | 与 `requires` 至少存在一个 | 具体 Skill/Adapter 名称字符串；当前 CLI 称为 legacy shortcut。 |
| `inputs` | 可选 | 输入映射数组。 |
| `inputs[].name` | 必填 | 字符串。 |
| `inputs[].from` | 必填 | 字符串；schema 不解释路径语法。 |
| `outputs` | 可选 | 预期输出数组。 |
| `outputs[].name` | 必填 | 字符串。 |
| `outputs[].artifactType` | 必填 | 字符串。 |

stage item 不允许表中之外的字段。schema 的 `anyOf` 允许同时写 `requires` 与 `uses`，但当前 Preflight 会把这种组合记录为 `AmbiguousStageBinding` 失败。

## Checkpoint 声明字段

| 字段 | 要求 | Schema 语义 |
| --- | --- | --- |
| `type` | 必填 | 字符串。 |
| `required` | 必填 | boolean。 |
| `after` | 可选 | stage id 字符串。 |
| `before` | 可选 | stage id 字符串。 |
| `reason` | 可选 | 字符串。 |

schema 不要求 `after` 或 `before`，也不禁止二者同时出现。当前 Preflight 不解析 checkpoint 位置；不要把解释器显示位置误认为关系验证。

## 合法示例

```yaml
apiVersion: huoban.dev/v1alpha1
kind: Flow
metadata:
  name: idea-to-spec-review
  generation: 2
spec:
  profileRefs:
    - name: huoban-open-source-project
  policyRefs:
    - name: safe-defaults
  stages:
    - id: sharpen-idea
      requires:
        capability: huoban.dev/designReview
        role: primary-review
      inputs:
        - name: idea
          from: user.prompt
      outputs:
        - name: review-notes
          artifactType: decision-notes
  checkpoints:
    - after: sharpen-idea
      type: humanApproval
      required: true
      reason: ConfirmDirectionBeforeSpec
```

对应 source example：`examples/flows/idea-to-spec-review.yaml`。

## 当前 CLI 行为

### Dry-run print

`huoban print --dry-run` 读取一个 Flow，生成 `mode: dryRun` 的 Run。它复制 Flow profile/policy refs，加入命令行 refs 和 bindings，并生成 `flowSpecHash`、`profileRefHash`、`policyRefHash`、`bindingHash`。

dry-run 只生成计划；它不执行 stage，也不自动运行 Preflight。

### Preflight

当前 Preflight 会：

- 解析 Flow ref 及请求的 generation。
- 解析 Flow 与 Run 合并后的 Profile/Policy refs。
- 报告重复 stage id 和重复 lifecycle handler id。
- 对 `requires` 按 role 查 Run binding，并检查 Skill/Adapter 是否提供 capability。
- 对 `uses` 要求 workspace 中恰好有一个同名 Skill 或 Adapter；它只支持 dry-run 与诊断，execute mode 会产生 `LegacyUseNotLocked`。
- 检查 Lifecycle Handler 的 stage selector、binding 和依赖。

`requires.role` 在 schema 中可选，但缺失 role 无法找到锁定 binding，当前 Preflight 会产生 `BindingMissing`。可执行的 capability-first Flow 应显式写 role。

## 验证边界

- schema 不要求 stage id 唯一；当前 Preflight 会检查重复。
- schema 不检查 input `from` 是否指向真实输出，也不检查 output name/type 兼容性。
- 当前 Preflight 不解析 checkpoint 的 `before`/`after` 或 reason。
- Flow 的 `status` 不保存运行进度；实际进度属于 Run。
- 当前仓库没有 Flow executor 或 Lifecycle dispatcher。

## 相邻概念边界

| 容易混淆 | 精确边界 |
| --- | --- |
| Flow / Run | Flow 是可复用版式；Run 锁定一次具体引用、binding 和 snapshot。 |
| stage `requires` / Requirement | stage `requires` 是 capability+role；`requirements[]` 是实现对象的环境依赖。 |
| Flow checkpoint / Checkpoint | Flow 声明位置；Checkpoint 顶层对象记录一次 Run 的具体决策边界。 |
| Flow output / Artifact | output 声明预期类型；Artifact 记录实际产物或证据。 |
| stage / Lifecycle Handler | stage 承载工作和状态改变；handler 只用于观察、上报和清理集成。 |

## 常见错误

- 在同一 stage 同时写 `requires` 和 `uses`，只因为 schema 可以通过。
- 省略 `requires.role` 却期望 Run 自动按 stage id 绑定。
- 把外部文件或可执行程序写进 stage `requires`；这些属于 Skill/Profile/Adapter 的 Requirement。
- 把 shell command、模型参数或 runner 配置写入 Flow。
- 声称 checkpoint 引用和 input/output 数据流已被当前 Preflight 验证。
- 认为 dry-run 已经执行 stage 或证明环境就绪。
